# 公開中継サーバー セットアップガイド

会場の採点システム（`server.js`）から確定得点だけを受け取り、インターネット経由の観覧者にリアルタイムで配信する「中継サーバー」です。観覧者側からは閲覧専用で、得点の入力・編集は一切できません。

**現在のデプロイ状況**: Render.com の `gym-public-relay` サービスとして稼働中。公開 URL は `https://gym-public-relay.onrender.com`（Service ID: `srv-d8pi1gugvqtc7397pqlg`）。GitHub リポジトリ `github.com/giantdiamidov-arch/gym-public-relay` の `main` ブランチへのプッシュで自動デプロイされます。

---

## 1. デプロイ先を決める（おすすめ: Render.com）

※以前は Glitch を案内していましたが、Glitch は 2025 年 7 月にアプリホスティングを終了しているため使えません。代わりに **Render.com** を使います。無料プランがあり、クレジットカード登録不要、Dockerfile などの専門知識も不要です（ただし GitHub リポジトリへのアップロードが必要です）。

### 事前準備: GitHub にこのフォルダをアップロードする

Git コマンドを使わなくても、ブラウザだけで完了できます。

1. https://github.com にアクセスし、無料アカウントを作成（未登録の場合）
2. 右上の「+」→「New repository」で新しいリポジトリを作成（例: `gym-public-relay`）。「Public」のままでOK
3. 作成後の画面で「uploading an existing file」というリンクをクリックし、中継サーバーのファイル一式（`server.js`・`package.json`・`public/` フォルダ）をドラッグ&ドロップでアップロードして「Commit changes」

### Render でのデプロイ手順

1. https://render.com にアクセスし、GitHub アカウントでログイン（無料）
2. ダッシュボードで「New +」→「Web Service」を選択
3. 先ほど作成した GitHub リポジトリ（`gym-public-relay`）を選択して連携
4. 設定項目:
   - **Build Command**: `npm install`
   - **Start Command**: `npm start`
   - **Instance Type**: Free
5. 「Environment」タブで環境変数を追加:
   ```
   RELAY_KEY = 好きな合言葉（例: gym2026secret）
   ```
   ※この値は、あとで会場側の `admin.html` にも同じ文字列を入力します。第三者に知られないようにしてください。**未設定のままにすると認証なしで誰でも `/push` を呼べる状態になるため、必ず設定してください。**
6. 「Create Web Service」をクリックすると数分でビルド・起動し、`https://（サービス名）.onrender.com` のような URL が発行されます

### 動作確認だけしたい / 短期間のみ使いたい場合: Railway

1 回のイベント運用だけなら、GitHub 連携が同じくらい簡単で初回登録時に無料クレジットが付与される **Railway**（https://railway.app）も候補です。ただしクレジットを使い切ると有料（月額制）に切り替わる点に注意してください。

## 2. 公開 URL を会場側の管理画面に設定する

会場の `admin.html` を開き、「🌐 公開ページ（クラウド中継）連携」セクションに以下を入力します:

- **中継サーバー URL**: `https://（サービス名）.onrender.com`
- **合言葉（Key）**: Render の環境変数に設定した `RELAY_KEY` と同じ文字列

保存すると、以後「得点確定」や「場内表示への反映」のたびに自動でこの中継サーバーへデータが転送されます。

## 5. スリープ対策（無料プランの制約）

Render の無料プランは**15 分間アクセスがないとスリープ**し、次のアクセス時に 50 秒以上の遅延が発生します。競技中に観覧者の最初のアクセスが遅れる原因になります。

### 対策A: 外部の死活監視サービスを使う（無料・推奨）

[UptimeRobot](https://uptimerobot.com) や [cron-job.org](https://cron-job.org) などの無料サービスから、`https://（サービス名）.onrender.com/health` を **5〜10 分おき**に定期アクセスするよう設定すると、スリープを防止できます。

### 対策B: 有料プランに切り替える

Render の Starter プラン（約 $7/月）以上に切り替えるとスリープがなくなります。コールドスタートの心配が完全になくなるため、本番運用での安心感が高まります。

### ⚠️ スリープ・再デプロイ後のデータ消失に注意

中継サーバーは受け取ったデータを**メモリ上にのみ保持**しており、スリープからの復帰や Render の再デプロイ後は得点データが初期値にリセットされます。上記の死活監視でスリープさせない運用が、このデータ消失リスクを避ける意味でも有効です。競技当日の開始前にデータが空の場合は、会場側で任意のスコアを確定操作すれば最新状態が中継サーバーに再送信されます。

## 6. 仕組みの補足

- `/push` エンドポイントだけが書き込み可能で、`x-relay-key` ヘッダーが `RELAY_KEY` 環境変数と一致しないと 401 を返します
- 観覧者が接続する WebSocket は閲覧専用ガードがかかっており、`PING` と `REQUEST_STATE` 以外のメッセージは無視されます。得点の改ざんはできない設計です
- 会場サーバーが送信するのは**確定済みスコア・VT 採用得点・一部の設定（E 審判人数・跳馬設定・花丸機能の有無）のみ**で、選手名簿の全件や入力途中のデータは送りません
- `common.js`（会場内用）と中継サーバーの `public/common.js` は内容が同一ですが、自動同期の仕組みはありません。E 平均計算などのロジックを修正した場合は両方に手動で反映してください
